## The Problem

**nVideo crashes with `0xC0000005` (access violation) when using HW-accelerated encode path.**

Example: `av1_nvenc` + `hwaccel: 'cuda'` crashes. SW encode (libx264, libsvtav1) works fine.

**Root cause identified:** In the transcode loop, HW-decoded frames are transferred from GPU to CPU memory via `av_hwframe_transfer_data()`, then passed to the encoder. But the encoder is configured with `hw_frames_ctx` (expecting GPU frames), causing a mismatch.

---

## Approaches Attempted (All Failed)

### Approach 1: Remove `hw_frames_ctx` from encoder, keep manual CPU transfer
**Theory:** Don't create `hw_frames_ctx` for encoder. Encoder receives CPU frames, handles its own GPU upload internally (like FFmpeg CLI default).
**Result:** Build succeeds, unit tests pass, but HW encode test fails with `"bad src image pointers"` from swscale. The `swsCtx` is being fed HW frames instead of SW frames — suggesting `av_hwframe_transfer_data()` is not working correctly or the frame format detection is wrong.

### Approach 2: Use FFmpeg filter graph with `hwdownload` filter
**Theory:** Let FFmpeg's filter graph handle HW→CPU transfer via `hwdownload,format=nv12` filter.
**Result:** Multiple failures:
- Buffer source configured with `yuv420p` but actual frames are `cuda` → format mismatch
- Buffer source configured with `cuda` → requires `hw_frames_ctx` to be set
- `sw_pix_fmt` on decoder doesn't force SW output (it's read-only)
- `av_buffersrc_parameters_set()` API was suggested by query_model but not successfully implemented

### Approach 3: Various combinations of the above
- With/without `hwdownload` filter
- With/without `pix_fmt` in buffer source args
- With/without `sw_pix_fmt` on decoder
- With/without scaling in filter graph vs `swsCtx`

**All combinations failed with different errors.**

## Key Technical Findings

1. **Decoder outputs `cuda` frames** (format=117) when `hwaccel=cuda` is set, regardless of `videoDecCtx->pix_fmt` after `avcodec_open2`
2. **`sw_pix_fmt` is read-only** — documented as "Set by libavcodec before calling get_format()"
3. **Buffer source filter requires `hw_frames_ctx`** when configured with HW pixel format — but setting it programmatically is complex
4. **`av_hwframe_transfer_data()` should work** for manual transfer, but swscale fails with "bad src image pointers" — possibly the transferred frame format doesn't match what swscale expects
5. **FFmpeg CLI handles this effortlessly** — the CLI code path is battle-tested, our C++ reimplementation is not

## Recommended Next Steps

### Option A: Use FFmpeg CLI for transcoding
**Pros:** Battle-tested, handles all HW/SW paths automatically  
**Cons:** Process spawning overhead, harder to get fine-grained progress  
**Effort:** Low — wrap `child_process.spawn('ffmpeg', ...)` with progress parsing

### Option B: Study FFmpeg CLI source code
**Pros:** Learn how `fftools/ffmpeg.c` handles HW acceleration properly  
**Cons:** Complex codebase, time-consuming  
**Effort:** High — but potentially the "right" solution

### Option C: Use `av_buffersrc_parameters_set()` properly
**Pros:** Clean filter-graph approach  
**Cons:** Requires understanding FFmpeg filter internals  
**Effort:** Medium — need to create `AVBufferSrcParameters`, set `hw_frames_ctx`, pass to buffer source

### Option D: Simplify nVideo to remove HW acceleration
**Pros:** Reliable, simple  
**Cons:** No GPU acceleration, slower transcoding  
**Effort:** Very low — just remove hwaccel code paths
